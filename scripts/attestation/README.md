# Popcorn SEV-SNP Attestation Verification

This directory contains standalone scripts to independently verify the cryptographic hardware attestation proofs generated by the Popcorn `attestor` sidecars running inside the AMD SEV-SNP Trusted Execution Environment (TEE).

## How It Works

When a Browser Pod boots up in the Popcorn cluster:
1. It runs on an AMD Epyc processor with SEV-SNP (Secure Encrypted Virtualization - Secure Nested Paging) enabled, ensuring memory encryption and state isolation from the hypervisor and host OS.
2. The `attestor` sidecar container queries the Kubernetes API from within the Pod to retrieve the exact immutable Image Digests (SHA256 hashes) of both the workload (`neko` Chromium browser) and itself.
3. It performs a Cosign verification check on those digests against the trusted Reclaim public key.
4. It computes a SHA-384 hash of the nonces, image digests, and the expected public key.
5. It calls the `/dev/sev-guest` hardware device to generate an attestation report, passing the computed hash into the 64-byte `REPORT_DATA` field.
6. The hardware returns the report alongside an ECDSA P-384 signature issued by the AMD Versioned Loaded Endorsement Key (VLEK).

The verification scripts in this folder automate the final **Client Verification** step, proving to you (the client) that the remote node is genuinely running the exact signed code you expect.

### What is the Attestation Report?

The SEV-SNP Attestation Report is a cryptographically signed data structure generated directly by the AMD hardware's Secure Processor inside the CPU. 
It cannot be spoofed by the host operating system, the hypervisor, or even AWS. 

It contains details about the execution environment (like the measurement of the launched VM) and, most importantly, a custom 64-byte field called `REPORT_DATA`. By injecting our own computed hash into `REPORT_DATA`, we securely bind arbitrary software data (like our image digests and nonces) to the hardware's root of trust.

### What are we verifying exactly?

When you run these verification scripts, you are performing a rigorous mathematical proof to guarantee that you are talking to a legitimate Popcorn node:

1. **Digest Transparency:** We read the returned Workload Image and Verifier Image digests from the API response.
2. **Freshness / Replay Prevention:** We validate that the returned Nonce matches the requested nonce (or the Pod Name for S3 archives), proving the proof was generated for this exact session and wasn't reused.
3. **Hardware Binding:** We recompute the `REPORT_DATA` SHA-384 hash locally using the exact same formula `H(workload_digest || verifier_digest || pubkey_hash || nonce)`.
4. **Report Integrity:** We decode the raw binary AMD SEV-SNP report and assert that the hardware-locked `REPORT_DATA` inside it *perfectly matches* our locally recomputed hash.
5. **Hardware Authenticity:** Finally, we parse the AMD VLEK X.509 Certificate chain and evaluate the ECDSA P-384 signature of the report. This mathematically proves that the report was genuinely generated by a genuine AMD CPU.

If all checks pass, you have absolute cryptographic certainty that the session is running the exact container images stated, and that it is fully isolated inside an AMD SEV-SNP Trusted Execution Environment.

## Verifying a Session

You can use the  **Node.js** implementation.

### Method 1: Node.js

Requires Node.js installed.

```bash
# Usage:
node scripts/attestation/verify_proof.js <SESSION_ID> [OPTIONAL_NONCE]

# Example:
node scripts/attestation/verify_proof.js session-1x
```


### Method 2: S3 Asynchronous Archive Verification (Node.js)

The Popcorn `attestor` cluster strictly uploads a hardware proof to an AWS S3 bucket immediately at boot. This allows you to verify the integrity of a session even after the cluster node shuts down. You can use these artifacts to manually re-verify the signatures using AMD's official tooling if desired.

Requires Node.js installed.

```bash
# Usage:
node scripts/attestation/verify_s3.js <POD_NAME> [AWS_REGION]

# Example:
node scripts/attestation/verify_s3.js browser-fleet-v4qln-74z59 us-east-2
```

This acts identically to the synchronous methods, but pulls `manifest.json`, the raw report, and the AMD VLEK credentials directly from the public S3 AWS bucket. Artifacts are saved to `/tmp/s3-proof-<POD_NAME>/`.
